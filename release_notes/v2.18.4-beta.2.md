## 🎯 Beta — False Manual Override Fix for Covers That Stay "Open" During Transit

Beta.2 fixes a false manual override triggered on covers that never emit `opening` or `closing` HA states while moving (common on French volet-roulant and some Zigbee rolling shutters).

### ✨ Features

- **Manual override gate logging (#280):** When a cover service call or state change is silently skipped because Manual Override Detection or Automatic Control is off, a DEBUG log is now emitted. Previously these were silent returns, making it impossible to tell from logs why manual override wasn't activating.
- **Manual override gate in diagnostics (#280):** `manual_toggle` and `enabled_toggle` (Automatic Control) booleans are now included in the diagnostics JSON under `configuration`, so the state of these gates is visible in every diagnostics dump.
- **Rename switch to "Manual Override Detection" (#280):** The Manual Override switch display name is now "Manual Override Detection" to better reflect its function. The entity `unique_id` is unchanged — no registry entries are affected.

### 🐛 Bug Fixes

- **False manual override on covers that stay "open" during transit (#285):** The transit progress check in `process_entity_state_change()` was gated on the cover emitting `opening` or `closing` HA state. Covers that stay `open` throughout transit (updating only `current_position`) skipped the check entirely — `wait_for_target` was cleared immediately, and the next position update was misidentified as a user move. The fix applies the direction/progress check based on position delta regardless of the HA state string. The step-motor mid-transit pause (Issue #186) is checked first so it still takes priority, and `stopped` state is excluded since it's an unambiguous halt signal.
