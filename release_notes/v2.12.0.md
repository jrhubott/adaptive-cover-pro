## 🎯 Release Summary

Major feature release with glare zone protection (floor-area shading), winter insulation mode (close non-sun covers for heat retention), irradiance accuracy fixes, override priority visualization, and improved configuration UI.

## ✨ Features

### Glare Zones (Issue #64) — Floor-Area Shading for Vertical Blinds

Define up to 4 named floor zones per vertical blind. Each zone is a circle on the floor (x/y position + radius in cm). The blind automatically lowers far enough to shade every active zone based on the current sun angle.

- **Per-zone configuration:** Set zone name, X/Y coordinates (cm), and protection radius
- **Per-zone switch entities:** Toggle zones on/off at runtime via automations or Home Assistant UI
- **Glare Active binary sensor:** Shows when glare protection is actively extending the blind position
- **Top-level options menu:** Glare Zones accessible without re-entering geometry configuration
- **Zero impact on existing setups:** Installations without zones configured work exactly as before

**Example workflow:** Configure a "Desk" zone at X=0, Y=200cm with 30cm radius. When the sun is in the blind's field of view, the blind automatically extends to shade that desk area. Toggle the zone off on weekends via automation.

### Winter Insulation Mode (Issue #29) — Close Non-Sun Covers for Heat Retention

In winter mode, automatically close covers that aren't exposed to direct sun, reducing heat loss through non-solar windows.

- **Smart seasonal behavior:** Opens sun-exposed covers for passive heating; closes shaded covers
- **Climate-aware:** Uses winter strategy to maximize energy efficiency
- **Automatic on cold days:** Works alongside existing winter opening for sunny windows
- **Reduces heating costs:** Minimize heat loss through windows not collecting solar gain

**How it works:** During winter, covers on non-sunny sides of the building close to an intermediate position. Covers facing the sun open to collect passive solar heat. Balances privacy, light, and energy efficiency.

## 🐛 Bug Fixes

### Issue #71 — Climate Mode Irradiance Accuracy

Irradiance sensors (pyranometers) and lux sensors are now properly respected in summer conditions.

- **Previously:** Irradiance was ignored when summer temperature threshold was exceeded
- **Now:** Low-light conditions (lux, irradiance, or non-sunny weather) trigger default position even during summer
- **Impact:** Covers respect sensor readings instead of blindly closing based on temperature alone

**Example:** If outdoor temp is 30°C (summer) but irradiance reads 200 W/m² (below threshold), covers stay open instead of closing.

**Technical:** Low-light checks moved before season logic in all climate state methods (`normal_with_presence`, `normal_without_presence`, `tilt_with_presence`, `tilt_without_presence`).

## 🏗️ Technical Details

### Glare Zones Architecture

**Pipeline Handler Implementation:**

- `GlareZoneHandler` (priority 45) — new handler in `pipeline/handlers/glare_zone.py`
- Only applies to `cover_blind` (vertical blinds); skipped for awnings/tilts
- Evaluates `PipelineSnapshot.active_zone_names` at each update
- Returns `PipelineResult` with `ControlMethod.GLARE_ZONE` when zone protection needed

**Zone Geometry Calculation:**

- Uses pure coordinate transforms (no HA imports) in `engine/covers/vertical.py`
- `glare_zone_effective_distance()` function computes required blind depth per zone
- Coordinate system: X (wall-perpendicular), Y (into room), Radius (protection circle)
- Edge case handling: elevation < 2° returns full window coverage
- Handles non-contiguous zone activation (e.g., zones 1 & 3 active, zone 2 skipped)

**Entity Creation:**

- Per-zone switch: `switch.cover_name_glare_zone_zone_name` (toggleable at runtime)
- Binary sensor: `binary_sensor.cover_name_glare_active` (on when extending blind)
- Existing installed covers unaffected when `enable_glare_zones=false`

### Winter Insulation Mode Architecture

**Climate State Enhancement:**

- New field in `ClimateCoverData` dataclass: `winter_close_insulation: bool`
- Four `ClimateCoverState` methods updated:
  - `normal_with_presence()`
  - `normal_without_presence()`
  - `tilt_with_presence()`
  - `tilt_without_presence()`
- Winter insulation position selected when all conditions met:
  - Temperature < `temp_low` (winter mode)
  - Sun elevation < threshold for window's azimuth (not sun-exposed)
  - `winter_close_insulation=true` in config

**Position Behavior:**

- Sun-exposed covers: Open fully for passive solar heat (existing winter logic)
- Non-sun covers: Close to `winter_close_position` (default: 60%) to reduce heat loss
- Automatic fallback: Returns to sun-calculated position if sun becomes available

### Climate Mode Irradiance Fix

**Logic Restructuring in `climate.py`:**

- Low-light checks (`lux_below_threshold` or `irradiance_below_threshold`) moved BEFORE season logic
- Prevents summer cooling when light levels insufficient
- Priority: Winter HEATING > Low-Light > Summer COOLING

**Methods Updated:**

```python
# Before: is_summer check first → ignored irradiance in summer
if is_summer:
    return POSITION_CLOSED
if low_light:  # Never reached in summer
    return default_position

# After: low-light check first → respects irradiance in all seasons
if low_light:
    return default_position
if is_summer:
    return POSITION_CLOSED
```

**Sensor Integration:**

- Reads from `irradiance_sensor` and `irradiance_threshold` (e.g., 500 W/m²)
- Reads from `lux_sensor` and `lux_threshold` (e.g., 10,000 lux)
- Weather provider respects `sunny_weather_states` list
- All three signals (irradiance, lux, weather) suppress glare control independently

### Performance Impact

**Glare Zone Calculations:**

- Per-zone distance computation: O(n) where n ≤ 4 zones
- Only runs when sun is in FOV (`cover.direct_sun_valid = true`)
- Coordinate transforms use numpy (vectorized) — negligible overhead
- Maximum 4 zones × ~2ms per zone = ~8ms per update cycle

**Winter Insulation:**

- Single boolean flag check in `ClimateCoverState` methods
- No performance impact when disabled
- Temperature/presence data already fetched by `ClimateProvider`

**Configuration Summary:**

- Narrative built once per config change (not every update cycle)
- Cached in `coordinator.data`
