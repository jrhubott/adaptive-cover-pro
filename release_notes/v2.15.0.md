## 🎯 Integration Kill Switch, Minimum Position Mode, and Comprehensive Bug Fixes

### ✨ Features

- **Integration Enabled master switch (#186):** A true kill switch for each ACP instance. When OFF, every outbound cover command is blocked — solar, climate, custom position, manual-override expiry, startup, end-of-window default, Force Override, and Weather Override. Motion and weather deferred timers are cancelled. Reconciliation state is cleared so no stale target replays on re-enable. Diagnostic sensors continue updating. Flip back ON and covers stay in place; the next natural update cycle resumes normally.
- **Immediate stop on disable (#186):** When Integration Enabled flips OFF, ACP immediately sends `cover.stop_cover` to any cover currently in motion (capability-checked).
- **Global services (#186):** Three new services for multi-instance / whole-home control, all accepting a `target:` block (entity, device, or area) to narrow scope:
  - `adaptive_cover_pro.emergency_stop` — stops every configured cover on all ACP instances, then disables.
  - `adaptive_cover_pro.integration_disable` — stops in-flight moves and disables.
  - `adaptive_cover_pro.integration_enable` — re-enables; covers stay in place.
- **Minimum Position Mode for Force/Weather/Custom Overrides (#178):** New toggle on Force Override, Weather Override, and all 4 Custom Position handlers. When enabled, the configured position acts as a floor — covers won't close below the threshold, but higher solar-tracked positions pass through. Default is off, preserving existing absolute-position behavior. Solves the window-contact-sensor use case without requiring template binary sensor workarounds.
- **All pipeline handlers run for sensor data (#182):** Refactored the pipeline to execute all priority handlers (not just the winner) so diagnostic sensors always report accurate data for every handler, even when lower-priority overrides are active. Improves observability without changing precedence logic.
- **Help text on config screens:** Added descriptive help text to the Light Sensors & Cloud and Temperature & Climate configuration screens.

### 🐛 Bug Fixes

- **False manual override on step-motor shades:** Step-motor shades move in pulses, briefly reporting `open` at intermediate positions between motor bursts. After grace period expiry this was incorrectly detected as a manual change, triggering a false override. Fixed: when a cover transitions from `opening`/`closing` to `open`/`closed` at a position that is not the target, the grace period is restarted instead of clearing `wait_for_target`, allowing the next motor pulse to proceed normally.
- **False manual override on covers with hardware startup delay (#186 / #172):** Covers with a 10–30+ second startup delay (motor engages slowly after receiving a command) fired their first state event before physically moving. The strict distance comparison treated equal distances as "not in transit" and cleared `wait_for_target`, causing the next position event to be flagged as a user move. The fix distinguishes a startup delay (`closed` → `opening`, position unchanged) from a genuine mid-transit stall (already `opening`, position unchanged).
- **Sunset position exempt from min/max position limits (#128):** A configured `min_position` setting could prevent covers from reaching their configured sunset position. `compute_default_position()` now exempts the sunset position from min/max clamping so covers always reach their sunset-hour destination.
- **Tilt cover position clamping for narrow FOV (#153):** `AdaptiveTiltCover.calculate_position()` could return degrees exceeding `max_degrees` (e.g. 101.4° with narrow FOV + high elevation). Added clamp to `[0, max_degrees]` before returning.
- **Manual Override duplicate unique_id collision (#154):** The Manual Override binary sensor and Manual Override switch both generated `{entry_id}_Manual Override` as their `unique_id`, causing HA entity registry collision. Fixed binary sensor to use the `key` field (`manual_override`) instead of the display name.
- **Diagnostics HTTP error — numpy types not JSON serializable (#155):** HA 2026.4.1+ tightened JSON serialisation and rejected `numpy` scalar types in entity state attributes. Calculation engine was leaking `numpy.bool_` and `numpy.float64`, breaking the REST API, state machine view, and the Download diagnostics button. Wrapped all `np.clip()`, `np.interp()`, and bitwise results in native `float()` / `bool()` at the calculation boundary.
- **Display rounding for sun angles and temperatures (#140):** Target Position displayed `1.0%` instead of `1%`; Sun Position displayed `1.00°` instead of `1°`. Rounding is now applied at presentation boundary only (no precision loss in calculations).
- **Start/end time not respected with climate and cloud suppression (#145):** `ClimateHandler` and `CloudSuppressionHandler` were missing the `in_time_window` gate. Covers moved based on temperature/cloud strategy regardless of the user's configured operational time window.
- **Manual override not detected during wait_for_target (#147):** When sun tracking sent a command (`wait_for_target=True`), a user manually moving the cover after the 5-second grace period was not detected as a manual override. Grace period expiry now correctly clears `wait_for_target`.
- **Duration format in config summary (#148):** Config summary showed raw dict `{'hours': 5, 'minutes': 0, 'seconds': 0}` instead of human-readable form. Now displays as `5 h`, `2 h 15 min`, etc.
