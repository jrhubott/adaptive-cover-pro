## 🎯 Comprehensive Bug Fixes, Minimum Position Mode, and Enhanced Testing

### ✨ Features

- **Minimum Position Mode for Force/Weather/Custom Overrides (#178):** Added a new "Minimum position mode" toggle to Force Override, Weather Override, and all 4 Custom Position handlers. When enabled, the configured position acts as a floor: covers won't close below the threshold, but higher solar-tracked positions pass through. Default is off, preserving existing absolute-position behavior for all existing users. Solves the window-contact-sensor use case without requiring template binary sensor workarounds.
- **All pipeline handlers now run for sensor data population (#182):** Refactored the pipeline to execute all priority handlers (not just the winner) so diagnostic sensors always report accurate data for every handler, even when lower-priority overrides are active. Improves observability without changing the precedence logic.

### 🐛 Bug Fixes

- **Sunset position now exempt from min/max position limits (#128, #184):** Previously, a configured `min_position` setting could prevent covers from reaching the configured sunset position. The `compute_default_position()` function now exempts the sunset position from min/max clamping so covers always reach their sunset-hour destination as configured.
- **Tilt cover position clamping for narrow FOV (#153):** `AdaptiveTiltCover.calculate_position()` could return degrees exceeding `max_degrees` (e.g. 101.4° for MODE1 with narrow FOV + high elevation). Added clamp to `[0, max_degrees]` before returning.
- **Manual Override duplicate unique_id collision (#154):** `binary_sensor.py` Manual Override and `switch.py` Manual Override both generated `{entry_id}_Manual Override` as their unique_id, causing HA entity registry collision. Fixed binary sensor to use the `key` field (`manual_override`) instead of the display name.
- **Diagnostics HTTP error — numpy not JSON serializable (#155):** HA 2026.4.1+ tightened JSON serialisation and rejects `numpy` scalar types in entity state attributes. Calculation engine was leaking `numpy.bool_` and `numpy.float64`, breaking the REST API, state machine view, and the Download diagnostics button. Wrapped all `np.clip()`, `np.interp()`, and bitwise comparison results in native `float()` / `bool()` at the calculation boundary.
- **Display rounding for sun angles and temperatures (#140):** Target Position sensor displayed `1.0%` instead of `1%`; Sun Position sensor displayed `1.00°` instead of `1°`. Implemented clean rounding at presentation boundary only (no precision loss in calculations): sun angles 1 decimal, temperatures 1 decimal, positions as integers.
- **Start/end time not respected with climate mode (#145):** `ClimateHandler` (priority 50) and `CloudSuppressionHandler` (priority 60) were missing the `in_time_window` gate added to `SolarHandler` in a prior refactor. Covers moved based on temperature strategy regardless of the user's configured operational time window. Added time-window check to both handlers.
- **Manual override not detected during grace period (#147):** When sun tracking sent a command (`wait_for_target=True`), a user manually moving the cover after the 5-second grace period expired was not detected as a manual override. Grace period expiry now clears `wait_for_target` so override detection can proceed normally.
- **Duration format in config summary (#146, #148):** Config summary showed raw `DurationSelector` dict `{'hours': 5, 'minutes': 0, 'seconds': 0}` instead of human-readable form. Added `_format_duration()` helper to display as `5 h`, `2 h 15 min`, etc.
