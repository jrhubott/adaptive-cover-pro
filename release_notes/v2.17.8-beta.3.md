## ⚠️ Beta Release v2.17.8-beta.3

This is a pre-release for testing. Please report any issues before the stable release.

## 🐛 Bug Fixes

### Fix glare zone priority — zones now override climate's solar tracking (Refs #231)
When climate mode was enabled, the glare zone handler (priority 45) was always outprioritized by the climate handler (priority 50), making glare zones completely ineffective even when a zone needed more protection. The fix raises glare zone priority to 55 (above climate, below cloud suppression). When no glare zones are active the handler falls through to climate as before.

**Note:** This is a deliberate behaviour change. Users with both climate mode and glare zones configured will now see glare zone positions take effect when a zone is active. If this causes unexpected interactions please report them on issue #231.

### Fix cloud suppression with unavailable weather entity (Refs #222)
When a weather entity becomes unavailable or is removed, cloud suppression was treating the unavailable state as "not sunny" and incorrectly staying active. An unavailable weather entity is now treated as sunny (skip check), consistent with how lux, irradiance, and cloud coverage readers handle unavailable entities.

### Fix per-user language in options menu (Refs #227)
The options flow menu was rendering labels in the HAOS system language instead of each user's profile language. Menu options are now passed as step IDs and translated client-side by the HA frontend, matching the pattern used elsewhere in the integration.

### Fix sun sensor start time rollover in time window (Refs #226)
When `sensor.sun_next_rising` or `sensor.sun_next_setting` are used as start time entities, the sensor flips to tomorrow's datetime once the event passes. This caused `start_time > end_time`, which logged an error and disabled cover tracking for the rest of the day. The fix normalises start time to today when the end time is still in the future.

### Decouple `force` from `is_safety` in PositionContext (Refs #223)
`force=True` (bypass delta/time gates) and `is_safety=True` (persist target across window boundaries) were conflated. The fix correctly classifies all four non-safety callers, preventing stale safety targets from suppressing reconciliation moves after a window closes.
