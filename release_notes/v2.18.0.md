## 🎯 Runtime Configuration Services + Glare Zone, Tilt Cover, and Sensor Reliability Fixes

### ✨ Features

- **Configure any setting from automations — Runtime Configuration Services (#221):** Fifteen new Home Assistant services under the `adaptive_cover_pro` domain let automations, scripts, and templates change integration settings at runtime without opening the Options UI. All changes persist across restarts and trigger a full integration reload automatically.

  Available services: `set_position_limits`, `set_sunset_sunrise`, `set_automation_timing`, `set_manual_override`, `set_force_override`, `set_custom_position`, `set_motion`, `set_light_cloud`, `set_climate`, `set_weather_safety`, `set_sun_tracking`, `set_blind_spot`, `set_interpolation`, `set_geometry`, `set_option`.

  Cross-field validation is enforced on each call: blind-spot distance ordering, temperature threshold ordering, custom position slot completeness, time-window mutual exclusion, and sunset dependency. Pass `null` for any field to clear it from options. The `set_option` service provides a lower-level escape hatch for any individual key not covered by the typed services.

### 🐛 Bug Fixes

- **Tilt-only covers now route correctly (#234):** When a cover entity only supports tilt operations (`SET_TILT_POSITION` but not `SET_POSITION`), Cover Pro was sending `open_cover` instead of `set_cover_tilt_position` because service routing relied on the config-level `sensor_type` rather than per-entity capabilities. This caused a position-verification retry loop firing approximately every two minutes because the device never acknowledged the position command. The fix routes service calls and position reads through tilt services when per-entity capabilities indicate tilt-only, regardless of the configured cover type. As a bonus, covers that report `STATE_UNKNOWN` (common with Z-Wave devices) but still expose `supported_features` now pass through the capability check correctly rather than being skipped entirely. (Thanks to @mscharwere for the fix.)

- **Glare zone now fires when climate mode is active with presence (#231):** In `ClimateCoverState.normal_with_presence()`, when the `GLARE_CONTROL` strategy was reached — occupants present, sun active, not a winter/summer/low-light case — the handler was returning a solar-tracking position instead of `None`. This blocked the pipeline from ever reaching the glare zone handler (priority 45, below climate at 50), making configured glare zones completely ineffective whenever climate mode and presence detection were both active. The fix returns `None` for the `GLARE_CONTROL` case, allowing the pipeline to fall through to the glare zone handler and then the solar handler as normal. Documentation priority references corrected from 55 to 45 (the code was always 45; only the docs were wrong). **Behaviour change:** users with climate mode, presence detection, and glare zones configured will now see glare zone positions take effect when the sun is active.

- **Cloud suppression now stays inactive when weather entity is unavailable (#222):** When a weather entity became unavailable or was removed, cloud suppression treated the unavailable state as "not sunny" and remained active, blocking solar tracking. An unavailable weather entity is now treated as sunny (check skipped), consistent with how lux, irradiance, and cloud coverage readers already handle unavailable entities.

- **Sun sensor time rollover no longer disables tracking (#226):** When `sensor.sun_next_rising` or `sensor.sun_next_setting` are used as time window start/end entities, each sensor flips to tomorrow's datetime immediately after the event passes. This produced `start_time > end_time`, which logged a time-window error and disabled cover tracking for the rest of the day. The fix normalises the sensor value to today when the paired end time is still in the future, and also handles rollover on the end-time sensor.

- **Options menu now renders in each user's language (#227):** The options flow menu was rendering labels in the HAOS system language rather than each user's individual profile language. Menu items are now passed as step IDs and translated client-side by the HA frontend, matching the pattern used elsewhere in the integration. Icon decorations are preserved.

- **Fix stale safety targets after window close (#223):** `force=True` (bypass delta/time gates) and `is_safety=True` (persist target across window boundaries in reconciliation) were conflated in `PositionContext`. All four non-safety callers are now correctly classified, preventing a stale safety-tagged target from suppressing reconciliation moves after the automation window closes.
