## 🎯 Sun Tracking, Glare Zone, and Climate Status Fixes

### ✨ Features

- **Glare zones now work independently of the sun tracking toggle (#238):**
  Disabling "Enable sun tracking" previously disabled both distance-based solar
  positioning *and* glare zone protection. Since glare zones have their own
  independent switch, the sun tracking toggle now only gates `SolarHandler`.
  `GlareZoneHandler` continues to run unconditionally based on its own
  configuration, time window, and cover type.

### 🐛 Bug Fixes

- **Sun sensor UTC times now converted to local timezone (#226):**
  When `sensor.sun_next_rising` / `sensor.sun_next_setting` were used as time
  window entities, the UTC ISO 8601 timestamps were stripped of timezone info and
  compared as wall-clock values. For non-UTC users this caused automation windows
  to open or close hours early. Times are now correctly converted to HA's
  configured local timezone before comparison.

- **Climate status sensor no longer shows `unknown` during glare control (#240):**
  Since v2.18.0, `ClimateHandler` returns `None` on the glare-control path to let
  `GlareZoneHandler` win. This also dropped `climate_data` from the winning
  result, causing the `climate_status` sensor to report `unknown` whenever
  intermediate-season glare control fired with climate mode enabled. `climate_data`
  is now propagated via a new `contribute()` hook regardless of which handler wins.

- **Config summary now shows time entities instead of static times (#243):**
  `_build_config_summary()` was always displaying the static `start_time` /
  `end_time` values even when `start_entity` / `end_entity` were configured,
  because the automation schema always populates static time fields with a
  `"00:00:00"` default. The summary now prefers entity references when present,
  matching the actual runtime behaviour.
