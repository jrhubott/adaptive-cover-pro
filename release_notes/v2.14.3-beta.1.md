## Bug Fixes

### Fix: Covers move at sunrise regardless of configured start time (#173, #170)

Covers were opening at sunrise even when a start time (or start time entity) was set to a later time. This affected all users who configured a start time later than sunrise.

**Root cause:** When the time-window gate was moved from `apply_position` into individual pipeline handlers (v2.13.6), `DefaultHandler` — the final fallback that always matches — was not given the gate. Any sun entity change before the configured start time would fall through to `DefaultHandler`, which returned the default position (e.g. 100% open) with no time-window check in the way.

**Fix:** Added the time-window guard in `async_handle_state_change` so covers are not repositioned outside the configured window. Safety handlers (force override, weather) still bypass this gate. Pipeline evaluation continues normally so diagnostics and sensor state remain accurate.

### Fix: Custom positions trigger manual override (#172)

Triggering a custom position via its binary sensor was incorrectly activating manual override, preventing the cover from resuming normal automatic behavior afterward.

**Root cause:** The integration-initiated cover move (sent by the custom position handler) was being detected as a user manual move, because the cover state change arrived before the command was registered as integration-originated.

**Fix:** Integration-initiated moves are now tracked and excluded from manual override detection.

### Fix: Horizontal cover crashes with NoneType on window_depth (#174)

Newly created horizontal covers crashed immediately with:

```
TypeError: '>' not supported between instances of 'NoneType' and 'int'
```

**Root cause:** The horizontal cover config flow schema does not collect `window_depth`, so the options dict stores `window_depth: None`. The service used `options.get(key, default)` which only applies the default when the key is _absent_ — not when it's present with value `None`.

**Fix:** Changed to `options.get(key) or 0.0`, consistent with the pattern already used for `sill_height` and other nullable fields.
