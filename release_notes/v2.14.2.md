### 🐛 Bug Fixes

**Fix #149 — Diagnostics HTTP error (numpy not JSON serializable)**

Users on Home Assistant 2026.4.1+ were hitting `TypeError: Type is not JSON serializable: numpy.bool` — breaking the REST API, the Developer Tools state view, and the Download Diagnostics button for all covers. This was caused by `numpy.bool_` and `numpy.float64` scalars leaking out of the geometric calculation engine into entity state attributes. All `np.clip()`, `np.interp()`, and bitwise comparison results are now explicitly cast to native Python `float()` / `bool()` at the calculation boundary.

**Fix #145 — Start/end time not respected when climate mode is enabled**

`ClimateHandler` (priority 50) and `CloudSuppressionHandler` (priority 60) were missing the `in_time_window` gate added to `SolarHandler` in a prior refactor. This caused covers with climate mode or cloud suppression enabled to open/close based on temperature strategy (e.g. full-open for winter heating, full-close for summer cooling) completely ignoring the user's configured operational start and end times. Both handlers now self-gate on `snapshot.in_time_window` and return `None` outside the window.

**Fix #147 — Manual override not detected during morning sun tracking**

When sun tracking sent a cover command (`wait_for_target=True`), a user manually moving the cover after the 5-second grace period expired was not being detected as a manual override. The `wait_for_target` flag was left set when grace expired but the cover hadn't yet reached the target, causing the override detection path to be skipped and the reconciliation cycle to re-send the automated command — fighting the user's manual move. The coordinator now clears `wait_for_target` when grace expires and the cover is not at the commanded target, allowing normal manual override detection to proceed.
