## ­¤Ä» Auto-Control Gate Fix, Event Timeline, input_boolean Support, and Diagnostics Improvements

Beta.3 fixes a critical bug where covers were commanded even when Automatic Control was switched off, and adds several diagnostics and configuration improvements.

### Ō£© Features

- **Event timeline in diagnostics (#294/#295):** The diagnostics JSON now includes an `event_timeline` ring buffer capturing up to 20 event types across the full cover lifecycle ŌĆö cover commands, pipeline evaluations, manual override transitions, `wait_for_target` lifecycle, and override state changes. `last_cover_action` gains 11 new fields including `target_source` (which caller sent the command), force/safety flags, integration state at call time, pipeline decision trace, and a `gates_evaluated` shadow check showing which gates `force=True` bypassed. The existing `manual_override_history` key is retained as a backward-compatible filtered view.

- **`enabled_handlers` attribute on decision_trace sensor (#297):** The `decision_trace` sensor now exposes an `enabled_handlers` list attribute naming the pipeline handlers that are actually configured for this cover instance. The companion Lovelace card uses this list to hide unconfigured handlers (e.g. a Force Override row when no force-override sensors are set). Configuration-based, not runtime-based: a handler appears in the list if its options are populated, regardless of whether its runtime switch is currently on.

- **`input_boolean` accepted in all boolean-gate selectors (#284):** Force override sensors, custom position sensors, motion sensors, weather sensors (rain, wind, severe), and the presence entity selector now accept `input_boolean` entities alongside `binary_sensor`. The presence entity selector is also narrowed from four domains (including `device_tracker` and `zone`) to two, since device_tracker and zone states are never `on`/`off`.

- **Manual override gate logging (#280):** When a cover service call or state change is silently skipped because Manual Override Detection or Automatic Control is off, a DEBUG log is now emitted. Previously these were silent returns, making it impossible to tell from logs why manual override wasn't activating.

- **Manual override gate in diagnostics (#280):** `manual_toggle` and `enabled_toggle` (Automatic Control) booleans are now included in the diagnostics JSON under `configuration`, so the state of these gates is visible in every diagnostics dump.

- **Rename switch to "Manual Override Detection" (#280):** The Manual Override switch display name is now "Manual Override Detection" to better reflect its function. The entity `unique_id` is unchanged ŌĆö no registry entries are affected.

### ­¤Éø Bug Fixes

- **Auto-control gate bypassed by force=True (#293):** When Automatic Control was switched off, the integration could still issue cover commands on certain internal paths that passed `force=True`. A cover commanded while the user was manually repositioning it would overwrite the user's position and leave `wait_for_target=True`, blocking subsequent manual moves. The fix ensures `apply_position` always respects the auto-control gate, and introduces `PositionContext.bypass_auto_control` as the sole sanctioned escape hatch (used only by the switch return-to-default one-shot). `async_handle_cover_state_change` also now correctly observes manual moves even when `auto_control=False`.

- **EventBuffer crashes on startup when buffer size is a float (#295 follow-up):** HA's `NumberSelector(SLIDER)` returns floats (e.g. `50.0`) for `CONF_DEBUG_EVENT_BUFFER_SIZE`. Passing a float to `deque(maxlen=...)` raises `TypeError`, preventing integration setup. The fix coerces at the `EventBuffer` boundary so any caller is safe.

- **Geometry menu button label mismatch (#289):** The options-flow menu button for the geometry step was labeled "Window Geometry" while the step title read "Cover Geometry". The label is now "Cover Geometry" across all three languages (en/de/fr).

- **False manual override on covers that stay "open" during transit (#285):** The transit progress check in `process_entity_state_change()` was gated on the cover emitting `opening` or `closing` HA state. Covers that stay `open` throughout transit (updating only `current_position`) skipped the check entirely ŌĆö `wait_for_target` was cleared immediately, and the next position update was misidentified as a user move. The fix applies the direction/progress check based on position delta regardless of the HA state string. The step-motor mid-transit pause (Issue #186) is checked first so it still takes priority, and `stopped` state is excluded since it's an unambiguous halt signal.
