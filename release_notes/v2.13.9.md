## 🎯 Major Features

### Config UX Simplification (beta.1)

Configuration experience improvements focused on reducing complexity. No runtime behavior changes — all improvements are UI/config flow only.

**Improvements:**
- **Quick Setup Flow** — New menu after cover type selection: Quick (5 screens) vs Full (all screens). Quick setup skips overrides, climate, and automation for faster initial configuration. All features remain accessible via Options menu.
- **Split Climate Screen** — Overcomplicated Climate screen divided into two focused screens:
  - ☁️ **Light Sensors & Cloud Suppression** — weather entity, lux, irradiance, cloud coverage, and suppression toggle  
  - 🌡️ **Temperature & Climate Mode** — indoor/outdoor temperature, presence, winter insulation
- **Weather Conditions Merged** — Sunny state selector folded into Light & Cloud screen, removing a standalone step
- **Position Map in Summary** — New section in config summary showing at a glance what position applies in each scenario
- **Options Menu Grouping** — Emoji prefixes visually organize sections: 📐 Setup, 📊 Position, ⏱️ Schedule, ☁️🌡️ Environment, 🛡️ Overrides, ⚙️ Admin
- **Reduced Switch Sprawl** — Lux, Irradiance, and Outside Temperature switches now hidden by default for new installations (entity registry preserves existing state)

### Custom Positions Pipeline Handler (beta.3)

A new priority-77 handler enables automation-driven cover positioning through up to 4 configurable binary sensors.

**How It Works:**
- Each sensor paired with a fixed cover position (0–100%)
- Sensors evaluated in order (1→4); first active sensor wins
- If all sensors are off, handler passes through silently
- Priority 77 sits between manual override (80) and motion timeout (75)

**Use Cases:**
- Scene-based positioning (Movie Mode → 5%, Bright Room → 50%)
- Schedule-driven automation (Morning → 30%, Evening → closed)
- Complex automation logic paired with Home Assistant automations

**Config Flow:**
- New "Custom Positions" step in both setup and options flows
- 4 independent sensor/position pairs, visually positioned before Motion Override
- Full translations in 13 languages

## 🔧 Technical Details

### Architecture

**New Handler:** `CustomPositionHandler` (priority 77)
- Evaluates ordered list of (entity_id, is_on, position) tuples
- Returns PipelineResult with decision trace on first active sensor match
- Passes through silently if no sensors configured or all off

**Data Types:** 
- `PipelineSnapshot.custom_position_sensors` — ordered list of sensor tuples
- `ControlMethod.CUSTOM_POSITION` — new control method enum value

**Coordinator:**
- `_read_custom_position_sensor_states()` — syncs binary sensor states each cycle
- Config constants: `CONF_CUSTOM_POSITION_SENSOR_1..4`, `CONF_CUSTOM_POSITION_1..4`

**Config Flow:**
- New `CUSTOM_POSITION_SCHEMA` with 4 sensor/position pairs
- Helper functions: `_position_slider()`, `_binary_sensor_selector()`
- Integrated into both setup and options flows

## 🐛 Known Limitations (v2.13.9)

### Custom Positions
- Sensors checked only on state changes; no polling fallback
- No visual indicator in config flow showing current sensor state
- Position values must be 0–100%; no per-sensor limits yet

### General
- Quick setup flow still shows all options in expanded mode (expected UX)
- Config summary does not auto-refresh when entities change (next update cycle)
